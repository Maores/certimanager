### [P0] Silent acceptance of variant column headers — cert dates dropped with no warning
- **Journey:** 06-import-nonstandard-columns.md
- **Persona:** dina-manager.md
- **Steps to reproduce:**
  1. Log in as `admin@test.local` and navigate to `/dashboard/import`.
  2. Upload `tests/agents/fixtures/pikoh-variant-cols.xlsx` (same 10 rows as `pikoh-happy`, headers: `ת.ז. | שם משפחה | שם פרטי | מצב | הסמכה | תאריך תוקף | רענון`).
  3. Review step shows: 10 parsed, 10 unique, 10 new, 0 skipped — **no warning anywhere on the page**.
  4. Commit with "ייבוא 10 עובדים".
  5. Navigate to `/dashboard/certifications` — all 10 new certs show status `לא ידוע` and dates `—`.
  6. Open employee detail for רחל ארז (700001111) — her cert has no issue/refresh dates.
- **Expected:** Either the parser's alias map covers these natural Hebrew variants (`תאריך תוקף`, `רענון`) so dates import correctly, or the review step visibly warns Dina that those columns were not recognized and dates will be blank.
- **Actual:** Neither happens. Parser at [src/lib/excel-parser.ts:289-290](src/lib/excel-parser.ts:289) only accepts exact strings `"תוקף תעודה"` and `"מועד רענון הבא"`. Variant headers resolve to column-index `-1` and dates are treated as absent. The review step shows cert pills with no date line; no banner; skipped-rows counter is 0. Summary screen reports "10 הסמכות נוצרו, 0 הסמכות דולגו". Dina has no visual cue that data was lost.
- **Screenshot:** media/2026-04-19-1554-06-certs-after.png
- **Classification:** bug

### [P1] Status column `מצב` silently falls back to default `פעיל` with no warning
- **Journey:** 06-import-nonstandard-columns.md
- **Persona:** dina-manager.md
- **Steps to reproduce:**
  1. Same upload as above. Variant file uses header `מצב` instead of `סטטוס`.
  2. On review step, all 10 rows show status `פעיל`.
- **Expected:** Either accept `מצב` as a status alias, or trigger the existing `statusWarning` UI ([src/components/import/review-step.tsx:92-99](src/components/import/review-step.tsx:92)) when the whole column is missing.
- **Actual:** `statusCol` resolves to `-1` ([excel-parser.ts:285](src/lib/excel-parser.ts:285) only looks for `סטטוס`/`סטאטוס`). `statusRaw` is `""` for every row; `normalizeStatus("", defaultStatus)` returns `"פעיל"`. `statusWarning` is never set. If the source file contained inactive or on-leave employees (`חל"ת`, `לא פעיל`), they'd all be silently flipped to `פעיל`.
- **Screenshot:** media/2026-04-19-1554-06-review-step.png
- **Classification:** bug

### Orchestrator notes

- **This run exactly validates the user-reported real-world concern.** The real manager uploads Pikoh files with non-canonical Hebrew headers; journey 06 proved that when he does, dates and statuses silently vanish and nobody tells him.
- **Precise code pointer:** [excel-parser.ts:282-290](src/lib/excel-parser.ts:282) is the alias table. Today it accepts:
  - Employee number: `מספר זהות`, `דרכון`, `ת.ז`, `תעודת זהות`
  - Status: `סטטוס`, `סטאטוס`
  - Everything else: canonical only (no aliases).
- **Two parallel fix strategies** (not mutually exclusive; can ship both):
  1. **Expand the alias table** (cheap, fast): add `תאריך תוקף`, `פקיעת תוקף`, `תוקף` for issue-date and `רענון`, `מועד חידוש`, `חידוש` for refresh-date. Same for `מצב` → status. This covers 80% of real-world variants at zero UX cost.
  2. **Add a "columns recognized" panel to the review step** (addresses the P2 ux): lists every header the parser saw, what it mapped it to, and highlights any unmatched columns. This is the permanent fix — it catches the next unknown variant too, not just the ones we pre-listed.
- The journey was paired with **dina-manager** (the persona most likely to encounter this in real use). Finding shapes match her biases precisely: she doesn't open DevTools, trusts what the UI shows, and would only discover the data loss when looking at a specific cert weeks later.
