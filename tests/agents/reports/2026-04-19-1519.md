### [P0] Non-xlsx file (txt renamed to .xlsx) silently accepted — advances to review step with 0 rows and NO error
- **Journey:** 04-import-dirty-data.md
- **Persona:** adversarial.md
- **Steps to reproduce:**

  3. Upload the file.
- **Expected:** Clear rejection (e.g. `שגיאה בקריאת הקובץ. ודאו שהקובץ תקין`) — remain on upload step with a red error banner.

Client validates only `file.name.endsWith('.xlsx')`; bogus content passes; the server-action error path is not being surfaced.

### Orchestrator notes

- **No crashes / no 500s.** All 6 bad inputs handled without stack traces or white screens. The bugs are **silent-acceptance** bugs, not robustness crashes. This is consistent with the persona's top bias: *"silent failure is worse than loud."*
- **One conceptual root cause across the P0/P1 set:** the import pipeline treats "row has data" as the only trigger for an error; "row has *bad* data" is routed through a happy-path-with-gaps. A fix would pre-flight each cell through a validator (date parser, Israeli-ID checksum, cert-type allowlist) and route rejects to `שורות שדולגו` with a reason, rather than letting the commit-step best-effort fill in gaps.
- **Real-world relevance:** the user has mentioned column-name variance in real Pikoh exports (e.g. `תז` vs `תעודת זהות`). That class of variance was NOT exercised by this journey because the fixture uses canonical column names. Suggest a follow-up journey that deliberately renames columns in a copy of the Pikoh fixture and reruns.
- **Preserved good behavior (no findings here):**
  - Past expiry date (`01/01/2020`) correctly renders as `פג תוקף`.
  - Short numeric IDs (`"123"`, `"456"`) correctly skipped with reason `מספר זהות לא תקין`.
  - Mixed RTL/LTR names render without layout corruption.
  - Parameterized queries held up (no SQL injection via the malicious last-name row).
  - Seed data preserved: 5 seed employees + 8 imported dirty-file employees = 13 total; 3 seed certs + 8 imported = 11 certs.
