### Root cause and fix (for audit trail)

- **Root cause:** staging Supabase was provisioned from `schema.sql` only. `migration_phase3.sql` and `migration_cert_types_v2.sql` were never applied, leaving three unique indexes and the entire `course_candidates` table missing. The import server action at [src/app/dashboard/import/actions.ts:189](src/app/dashboard/import/actions.ts:189) and [:233](src/app/dashboard/import/actions.ts:233) references these as `onConflict` targets, causing Postgres `42P10` on every commit. Production has these migrations applied and works normally.
- **Fix (defense-in-depth):** Rolled all upsert-critical DDL (three `CREATE UNIQUE INDEX IF NOT EXISTS` + the full `CREATE TABLE IF NOT EXISTS course_candidates` with its three ordinary indexes) into [tests/agents/fixtures/seed.sql](tests/agents/fixtures/seed.sql) before the TRUNCATE step. Each run of `test:agents:reset` now self-heals any schema drift on the target Supabase. The production database is not touched by this change — reset-db refuses to run unless `SUPABASE_ENV=staging`.
- **Audit:** `scripts/check-staging-indexes.ts` probes all four upsert conflict targets used by the app (employees / cert_types / certifications / course_candidates). Post-fix: all four return `OK`. Before-fix: three returned `42P10`, one returned `PGRST205` (table missing).
- **Orchestrator notes:** the original Run 2 had 3 UX findings unrelated to the P0 (row-click affordance, refresh-drops-state, browser-Back-exits-wizard). Those were NOT addressed by this fix and remain open — deferred to the post-harness triage pass per orchestrator policy.
